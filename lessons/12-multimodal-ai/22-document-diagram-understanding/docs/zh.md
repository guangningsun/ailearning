# 文档与图表理解

> 文档不是照片。PDF、科学论文、发票或手写表单都有布局、表格、图表、脚注、页眉和语义结构，这些是普通图像理解无法捕捉的。VLM 时代之前的堆栈是一条流水线：Tesseract OCR + LayoutLMv3 + 表格抽取启发式。VLM 浪潮用 OCR-free 模型取代了它——Donut（2022）、Nougat（2023）、DocLLM（2023）——直接输出结构化标记。到 2026 年，前沿变成了"把页面图像以 2576px 原生分辨率喂给 Claude Opus 4.7"，结构化标记输出免费获得。本课解读文档 AI 的三个时代。

**类型：** 构建型
**语言：** Python（标准库，支持布局的文档解析器骨架）
**前置条件：** 阶段 12 · 05（LLaVA）、阶段 5（NLP）
**时间：** 约 180 分钟

## 学习目标

- 解释文档 AI 的三个时代：OCR 流水线、OCR-free、VLM 原生。
- 描述 LayoutLMv3 的三个输入流：文本、布局（bbox）、图像块，统一掩码。
- 对比 Donut（OCR-free，图像→标记）、Nougat（科学论文→LaTeX）、DocLLM（支持布局的生成式）、PaliGemma 2（VLM 原生）。
- 为新任务选择文档模型（发票、科学论文、手写表单、中国收据）。

## 问题

"理解这个 PDF"实际上非常困难。信息分布在：

- 文本内容（90% 的信号）。
- 布局（页眉、脚注、侧边栏、双栏格式）。
- 表格（行、列、合并单元格）。
- 图表和示意图。
- 手写批注。
- 字体和排版（标题 vs 正文）。

原始 OCR 会倾倒文本而丢失其余部分。一个关心发票的系统需要知道"Total: $1,245"来自右下角，而不是脚注。

## 概念

### 时代 1 — OCR 流水线（2021 年之前）

经典堆栈：

1. PDF → 每页图像。
2. Tesseract（或商业 OCR）提取文本，包含每个词的边界框。
3. 布局分析器识别块（页眉、表格、段落）。
4. 表格结构识别器解析表格。
5. 领域规则 + 正则表达式提取字段。

适用于干净的打印文本。手写、倾斜扫描、复杂表格、非英语脚本都会出错。每个失败模式都需要自定义异常路径。

### TrOCR（2021）

TrOCR（Li 等，arXiv:2109.10282）用变换器编码器-解码器取代了 Tesseract 的经典 CNN-CTC，在合成 + 真实文本图像上训练。手写和多语言文本上干净利落地胜出。仍然是流水线（检测器→TrOCR→布局），但 OCR 步骤显著改善。

### 时代 2 — OCR-free（2022-2023）

第一批 OCR-free 模型说：完全跳过检测，直接将图像像素映射到结构化输出。

Donut（Kim 等，arXiv:2111.15664）：
- 编码器-解码器变换器，编码器是 Swin-B。
- 输出是 JSON（表单理解）、markdown（摘要）或任何特定任务的模式。
- 无 OCR，无布局，无检测。

Nougat（Blecher 等，arXiv:2308.13418）：
- 专门针对科学论文训练。
- 输出是 LaTeX / markdown。
- 处理公式、多栏布局、图表。
- 每个 arXiv 解析器都在调用的模型。

这些是专精模型，不是通才。Donut 处理科学论文会失败；Nougat 处理发票会失败。

### LayoutLMv3（2022）

一条不同的路线。LayoutLMv3（Huang 等，arXiv:2204.08387）保留 OCR 但增加了布局理解：

- 三个输入流：OCR 文本标记、每标记 2D 边界框、图像块。
- 跨三种模态的掩码训练目标（掩码文本、掩码块、掩码布局）。
- 下游任务：分类、实体抽取、表格问答。

LayoutLMv3 是基于 OCR 的文档理解的巅峰。在表单和发票上表现强劲。需要上游 OCR。在标准化文档基准上具有最佳的前 VLM 准确率。

### DocLLM（2023）

DocLLM（Wang 等，arXiv:2401.00908）是 LayoutLM 的生成式兄弟。在布局标记条件下生成自由形式答案。更适合文档问答；仍然依赖 OCR 输入。

### 时代 3 — VLM 原生（2024 年以后）

2024 年的 VLM 已经足够好，可以完全取代流水线。将全页图像以高分辨率喂给 VLM，提出问题，获得答案。

- LLaVA-NeXT 336-tile AnyRes 适用于小文档。
- Qwen2.5-VL 动态分辨率原生处理 2048+ 像素。
- Claude Opus 4.7 支持 2576px 文档。
- PaliGemma 2（2025 年 4 月）专门针对文档 + 手写训练。

VLM 原生与 OCR 流水线之间的差距迅速缩小。到 2026 年，VLM 原生在以下方面胜出：

- 场景文本（手写 + 打印，混合脚本）。
- 带合并单元格的复杂表格。
- 嵌入文本中的数学公式。
- 带文本标注的图表。

OCR 流水线仍在以下方面胜出：

- 大规模纯扫描工作负载，每页延迟很重要。
- 流水线可靠性（确定性失败 vs VLM 幻觉）。
- 需要可审计 OCR 输出的受监管环境。

### Claude 4.7 / GPT-5 前沿

在 2576 像素原生输入下，前沿 VLM 以接近人类的准确率进行文档理解。2026 年初的基准数据：

- DocVQA：Claude 4.7 ~95.1，PaliGemma 2 ~88.4，Nougat ~77.3，流水线 LayoutLMv3 ~83。
- ChartQA：Claude 4.7 ~92.2，GPT-4V ~78。
- VisualMRC：Claude 4.7 ~94。

闭源模型差距主要来自分辨率和基础 LLM 规模。7B 开源模型差几分但在追赶。

### 数学公式与 LaTeX 输出

科学论文需要精确的 LaTeX 公式输出。Nougat 专门为此训练。使用 LaTeX 目标训练的 VLM（Qwen2.5-VL-Math、Nougat 衍生模型）产生可用的 LaTeX。没有显式 LaTeX 训练，VLM 产生可读但不精确的转录。

2026 年科学论文流水线：先用 Nougat 处理 PDF，再用 VLM 处理棘手页面。

### 手写

仍然是最难的子任务。混合打印 + 手写（医生笔记、填写表单）是 OCR 流水线仍然以成本胜出 VLM 的地方。纯手写 VLM 正在改善（Claude 4.7、PaliGemma 2）。

### 2026 配方

对于新的文档 AI 项目：

- 大规模纯打印发票：LayoutLMv3 + 规则，成本效益高。
- 混合文档（科学 + 手写 + 表单）：VLM 原生（PaliGemma 2 或 Qwen2.5-VL）。
- 完整 arXiv 摄入：Nougat 处理数学，VLM 处理图表。
- 受监管场景：OCR 流水线 + VLM 验证器交叉检查。

## 使用它

`code/main.py`：

- 一个玩具级布局感知分词器：给定（文本，bbox）对，产生 LayoutLMv3 风格的输入。
- 一个 Donut 风格的任务模式生成器：表单的 JSON 模板。
- 每页跨 OCR 流水线、Donut、Nougat 和 VLM 原生的令牌预算比较。

## 交付它

本课产出 `outputs/skill-document-ai-stack-picker.md`。给定一个文档 AI 项目（领域、规模、质量、受监管），在 OCR 流水线、OCR-free 专家和 VLM 原生之间选择。

## 练习

1. 你的项目每天处理 1000 万张发票。哪个堆栈在不失准确率的情况下使每页成本最低？

2. 为什么 LayoutLMv3 在表单问答上优于纯 CLIP-VLM，但在场景文本上不如？bbox 流放弃了什么？

3. Nougat 生成 LaTeX。提出一个 VLM 原生输出在 LaTeX 保真度上胜过 Nougat 的测试用例，以及 Nougat 胜出的用例。

4. 阅读 PaliGemma 2 论文（Google，2024）。与 PaliGemma 1 相比，提升文档准确率的关键训练数据添加是什么？

5. 设计一个受监管安全的混合方案：OCR 流水线作为主要方案，VLM 作为次要交叉检查。你如何解决分歧？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------------|------------------------|
| OCR 流水线 | "Tesseract 风格" | 分阶段堆栈：检测→OCR→布局→规则；确定性，脆弱 |
| OCR-free | "Donut 风格" | 跳过显式 OCR 的图像到输出变换器；单一模型 |
| 支持布局 | "LayoutLM" | 输入包含每标记 bbox 坐标；跨模态统一掩码 |
| VLM 原生 | "前沿 VLM" | 将页面图像直接喂给 Claude/GPT/Qwen VLM 高分辨率；无流水线 |
| DocVQA | "文档基准" | 文档 VQA 标准；被引用最多的分数 |
| 标记输出 | "LaTeX / MD" | 结构化输出格式而非自由形式文本；支持下游自动化 |
