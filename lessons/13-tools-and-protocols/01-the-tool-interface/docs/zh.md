# 工具接口 —— 为什么智能体需要结构化输入/输出

> 语言模型生成 token。程序执行动作。两者之间的鸿沟就是工具接口：一份让模型请求动作、让宿主执行动作的契约。2026 年的技术栈——OpenAI、Anthropic 和 Gemini 的 function calling；MCP 的 `tools/call`；A2A 的 task parts——都是同一个四步循环的不同编码。本课命名这个循环，并展示运行它所需的最少机制。

**类型：** 学习型
**语言：** Python（标准库，无需 LLM）
**前置条件：** 阶段 11（LLM 补全 API）
**时间：** 约 45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM 本身无法对现实世界采取行动。
- 画出四步工具调用循环（描述 → 决定 → 执行 → 观察）并指出每步由谁负责。
- 将工具描述写为三个部分：名称、JSON Schema 输入和一个确定性执行函数。
- 区分纯工具和副作用工具，并说明这种区分对安全为何重要。

## 问题

LLM 输出的是下一个 token 的概率分布。这就是全部的输出表面。如果你问一个聊天模型"现在班加罗尔的天气怎么样"，它能写出一个看似合理的句子，但无法拨打天气 API。这个句子可能是巧合猜对了，也可能是三天前的旧信息。

工具接口的存在就是为了弥合这个鸿沟。宿主程序——你的智能体运行时、Claude Desktop、ChatGPT、Cursor 或自定义脚本——向模型公布一份可调用工具的列表。当模型决定需要采取行动时，它会发出一条结构化载荷，指名一个工具及其参数。宿主解析该载荷，真正运行该工具，然后将结果反馈回去。循环继续，直到模型决定不再需要调用。

这份契约的第一个版本于 2023 年 6 月随 OpenAI 的"functions"参数发布。Anthropic 在 Claude 2.1 中引入了 `tool_use` 块。Gemini 几个月后添加了 `functionDeclarations`。每个提供商现在都暴露相同的形状：输入 JSON-Schema 类型化工具列表，输出 JSON 载荷工具调用。模型上下文协议（2024 年 11 月）将这份契约泛化，使一个工具注册表可为所有模型服务。A2A（2026 年 4 月，v1.0）在同一个原语上叠加了智能体到智能体委托的相同机制。

四步循环是所有这些底下的不变式。阶段 13 的其余内容都是对它的详细阐述。

## 概念

### 第一步：描述

宿主用三个字段声明每个工具。

- **名称。** 一个稳定的、机器可读的标识符。用 `get_weather`，而不是"天气那个东西"。
- **描述。** 一段自然语言的简要说明。"当用户询问特定城市当前天气时使用。请勿用于历史数据。"
- **输入模式。** 一个描述工具参数的 JSON Schema 对象（2020-12 版）。

模型收到列表。现代提供商使用特定于提供商的模板将这些声明序列化到系统提示中，因此你作为调用者只需处理结构化形式。

### 第二步：决定

根据用户的消息和可用工具，模型选择三种行为之一。

1. **直接回答** 文本。不调用工具。
2. **调用一个或多个工具。** 发出结构化调用对象。在 `parallel_tool_calls: true`（OpenAI 和 Gemini 默认，Anthropic 可选开启）下，模型可以在一次轮次中发出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产生一个类型化的 `refusal` 块而不是调用。

工具调用载荷有三个稳定字段：调用 `id`、工具 `name` 和 JSON `arguments` 对象。id 的存在是为了让宿主能够将后续结果与特定调用关联起来——当并行调用乱序返回时，这一点很重要。

### 第三步：执行

宿主收到调用，根据声明的模式验证参数，然后运行执行器。无效参数意味着模型幻觉出一个字段或使用了错误的类型——这是在弱模型上非常常见的失败模式。生产宿主在遇到无效参数时有三种选择：快速失败并将错误呈现给模型、用约束解析器修复 JSON、或在提示中包含验证错误后重试模型。

执行器本身是普通代码。Python、TypeScript、shell 命令、数据库查询。它产生一个结果，结果通常是字符串，但可以是任何 JSON 值或结构化内容块（MCP 中的文本、图像或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主将工具结果追加到对话中（作为具有匹配 `id` 的 `tool` 角色消息），然后重新调用模型。模型现在在上下文中有了工具输出，可以产生最终答案或请求更多调用。这继续进行，直到模型停止发出调用或宿主达到安全限制的迭代次数。

### 信任分割

工具分为两种与安全相关的类型。

- **纯工具。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以安全地 speculative 调用。
- **有后果的工具。** 改变状态、花钱、触碰用户数据。`send_email`、`delete_file`、`execute_trade`。必须加以限制。

Meta 2026 年的"两条规则"智能体安全准则规定，单次轮次最多可组合以下三项中的两项：不受信任的输入、敏感数据、有后果的行动。工具接口正是你执行该规则的地方——通过拒绝调用、要求用户确认或升级作用域。详见阶段 13 · 15 安全章节和阶段 14 · 09 智能体级权限策略。

### 循环位于何处

| 上下文 | 谁来描述 | 谁来决定 |谁来执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | 通过 MCP 客户端的 LLM | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用方智能体 | 被调用智能体 |
| Web 浏览器（函数调用智能体） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

到处都是同样的四步。列名变了；结构没变。

### 为什么不让模型直接提示发出 JSON？

"让模型用 JSON 回复"是函数调用之前的模式。在前沿模型上它有大约 5% 到 15% 的失败率，在更小的模型上失败率更高。失败模式包括缺少大括号、尾随逗号、幻觉字段和错误类型。然后你需要 JSON 修复、一次重试或一个约束解码器。

原生函数调用更好，有三个原因。首先，提供商在确切的调用形状上端到端训练模型，因此在严格模式下有效 JSON 率攀升至 98% 到 99%。其次，调用载荷位于自己的协议槽中，而不是在自由文本内部——因此工具调用永远不会泄漏到用户可见的回复中。第三，提供商通过约束解码强制执行模式合规（OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）。输出保证可验证。

### 断路器

当模型停止发出调用或宿主达到最大轮次数时，循环终止。生产宿主将其设置为 5 到 20 轮之间。超过这个数字，你几乎肯定陷入了一个模型无法退出的循环。Claude Code 默认为 20；OpenAI Assistants 为 10；Cursor 的智能体模式为 25。

替代方案——无界循环——每六个月就会出现在"智能体一夜之间在 API 调用上花了 400 美元"的事后分析中。不要在没有界限的情况下发布。

## 使用它

`code/main.py` 运行四步循环，无需 LLM。一个伪造的"决策者"函数通过模式匹配用户消息来模拟模型；执行器、模式验证器和观察步骤工具是真实的。运行它来查看带有可打印中间状态的完整请求/响应流程，然后用后续课程中的任何真实提供商替换伪造的决策者。

要看的内容：

- 工具注册表每个工具持有三个字段：名称、描述、模式和执行器引用。
- 验证器是一个最小的 JSON Schema 子集（类型、必需、枚举、最小/最大），仅用标准库编写。阶段 13 · 04 发货一个更完整的。
- 循环将迭代次数限制在五个。生产智能体正是需要这种断路器。

## 发货它

本课产出 `outputs/skill-tool-interface-reviewer.md`。给定一个草稿工具定义（名称 + 描述 + 模式 + 执行器大纲），该技能审查它是否符合循环适用性：名称是否机器稳定、描述是否完整、模式是否正确使用 JSON Schema 2020-12，以及纯工具与有后果工具的分类是否明确。

## 练习

1. 在 `code/main.py` 中添加一个名为 `get_stock_price(ticker)` 的第四个工具。将其描述写为"当用户按 ticker 询问当前股票价格时使用。请勿用于历史价格或市场摘要。"运行工具并确认伪造的决策者将提到 ticker 的查询路由到新工具。

2. 破坏模式验证器。传递一个其 `arguments` 对象缺少必需字段的调用，并确认宿主在执行前拒绝它。然后传递一个带有额外未知字段的调用。决定：宿主应该拒绝还是忽略？用安全论点证明你的选择。

3. 将工具中的每个工具分类为纯工具或有后果工具。向需要它的注册表条目添加 `consequential: true` 标志，并更改循环以在选择有后果工具时打印"将与用户确认"一行。这是每个生产宿主需要的确认门控的形状。

4. 在纸上画出四步循环，填写上面你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT 或自定义堆栈）的提供商列。用阶段 13 · 06 中的 MCP 特定变体进行交叉引用。

5. 自上而下阅读 OpenAI 的函数调用指南。找出在请求中存在但不在这里呈现的四步循环中的一个字段。解释它添加了什么以及为什么它是方便的而不是必需的。

## 关键术语

| 术语 | 大家怎么说的 | 实际含义 |
|------|----------------|------------------------|
| 工具 (Tool) | "模型可以调用的东西" | 名称 + JSON-Schema 类型输入 + 执行器函数的三元组 |
| 函数调用 (Function calling) | "原生工具使用" | 提供商级 API 支持发出结构化工具调用而不是散文 |
| 工具调用 (Tool call) | "模型请求行动" | 由模型发出的带有 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果 (Tool result) | "工具返回了什么" | 执行器的输出，包装在带有匹配 id 的 `tool` 角色消息中 |
| 并行工具调用 (Parallel tool calls) | "一次多个调用" | 一次模型轮次中的多个调用对象，独立且可按 id 排序 |
| 严格模式 (Strict mode) | "保证 JSON" | 约束解码，强制模型的输出根据声明的模式进行验证 |
| 纯工具 (Pure tool) | "只读工具" | 无副作用；可以安全地重新运行 |
| 有后果的工具 (Consequential tool) | "行动工具" | 改变外部状态；需要门控、审计或用户确认 |
| 四步循环 (Four-step loop) | "工具调用循环" | 描述 → 决定 → 执行 → 观察 |
| 宿主 (Host) | "智能体运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
