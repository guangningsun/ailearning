# LLM 路由层 — LiteLLM、OpenRouter、Portkey

> 提供商锁定代价高昂。不同的工具调用工作负载适配不同的模型。路由网关提供统一的 API 表面、重试机制、故障转移、成本追踪和安全护栏。2026 年主导的三种架构：LiteLLM（开源自托管）、OpenRouter（托管 SaaS）、Portkey（生产级，2026 年 3 月开源）。本课给出决策标准，并实现一个标准库路由网关。

**类型：** 学习型
**语言：** Python（标准库、路由 + 故障转移 + 成本追踪器）
**前置条件：** 阶段 13 · 02（函数调用）、阶段 13 · 17（网关）
**时间：** 约 45 分钟

## 学习目标

- 区分自托管、托管和生产级路由方案。
- 实现故障转移链，按优先级顺序在提供商故障时重试。
- 跨提供商追踪每请求成本和 Token 使用量。
- 根据给定的生产约束条件在 LiteLLM、OpenRouter 和 Portkey 之间做出选择。

## 问题

提供商路由重要的场景：

1. **成本。** Claude Sonnet 费用是 Haiku 的 3 倍。分流任务用 Haiku 就够了；综合任务值得用 Sonnet。按请求路由。

2. **故障转移。** OpenAI 故障一小时。每个请求都失败。你想在不重新部署的情况下自动回退到 Anthropic。

3. **延迟。** 实时聊天 UI 需要快速的首 Token 时间。批量摘要器不需要。按延迟 SLA 路由。

4. **合规。** 欧盟用户必须留在欧盟地区。按区域路由。

5. **实验。** 在同一工作负载上对两个模型做 A/B 测试。按测试桶路由。

每个集成都手工编写这些逻辑是重复的。路由网关提供统一的 OpenAI 兼容 API，其余由网关处理。

## 概念

### OpenAI 兼容代理形状

大家都讲 OpenAI 形状。路由网关对外暴露 `/v1/chat/completions`，接收 OpenAI schema，在内部代理到 Anthropic / Gemini / Cohere / Ollama / 任意后端。客户端不关心。

### 模型别名

不用 `claude-3-5-sonnet-20251022`，你的代码写 `our_smart_model`。网关将别名映射到真实模型。当 Anthropic 发布 Claude 4 时，你在服务端改别名配置；你的代码一动不动。

### 故障转移链

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

网关在配置中定义这个。重试计入预算，防止故障转移级联导致成本爆炸。

### 语义缓存

相同或几乎相同的提示词直接命中缓存，而不是发往提供商。在重复的 Agent 循环中可节省 30% 到 60%。键基于 Embedding；几乎相同的提示词共用一个缓存槽。

### 护栏

网关层面：

- **PII 脱敏。** 发送提示词之前用正则或 ML 方式过滤。
- **策略违规。** 拒绝包含禁止内容的提示词。
- **输出过滤器。** 清洗泄露内容的回复。

Portkey 和 Kong 都提供自主方案的护栏。LiteLLM 将护栏作为可选项。

### 按密钥的速率限制

一个 API 密钥 = 一个团队。按密钥预算防止某个团队耗尽共享配额。大多数网关支持此功能。

### 自托管 vs 托管的权衡

| 因素 | LiteLLM（自托管） | OpenRouter（托管） | Portkey（生产级） |
|--------|----------------------|----------------------|----------------------|
| 代码 | 开源，Python | 托管 SaaS | 开源（2026 年 3 月）+ 托管 |
| 部署 | 部署一个代理 | 注册账号 | 两者都行 |
| 提供商 | 100+ | 300+ | 100+ |
| 计费 | 用你自己的密钥 | OpenRouter 积分 | 用你自己的密钥 |
| 可观测性 | OpenTelemetry | 仪表盘 | 完整 OTel + PII 脱敏 |
| 适用场景 | 想要完全控制数据的团队 | 快速原型开发 | 需要合规范的生产环境 |

当你有 SRE 团队并想要数据主权时，LiteLLM 胜出。当你想要单一订阅且不想维护基础设施时，OpenRouter 胜出。当你开箱即需要护栏和合规支持时，Portkey 胜出。

### 成本追踪

每个请求携带 `provider`、`model`、`input_tokens`、`output_tokens`。乘以每模型每 Token 价格（从网关维护的定价表中获取）。按用户 / 团队 / 项目聚合。

### MCP 加路由

网关可以同时路由 LLM 调用和 MCP 采样请求。当采样请求的 modelPreferences 指定偏好某个模型时，网关翻译到正确的后端。这就是阶段 13 · 17（MCP 网关）和本课路由网关有时合并为一个服务的原因。

### 路由策略

- **静态优先级。** 列表中排第一；故障时回退。
- **负载均衡。** 轮询或加权。
- **成本感知。** 挑选满足延迟 / 质量要求的最便宜模型。
- **延迟感知。** 挑选过去 N 分钟内最快的模型。
- **任务感知。** 提示词分类器将编码任务路由到一个模型，将摘要任务路由到另一个。

## 使用

`code/main.py` 用约 150 行代码实现了一个路由网关：接收 OpenAI 形状的请求，翻译为每个提供商的存根，运行优先级故障转移链，追踪每请求成本，并在输入上应用 PII 脱敏。用三个场景运行：正常请求、触发故障转移的主提供商中断、PII 泄露被脱敏拦截。

需要关注的地方：

- `ROUTES` 字典：别名 -> 按优先级排序的具体提供商列表。
- 故障转移循环在 5xx 上重试。
- 成本追踪器将 Token 使用量乘以每模型费率。
- PII 脱敏器在转发前清洗 SSN 形状的模式。

本课产出 `outputs/skill-routing-config-designer.md`。给定一个工作负载配置（延迟、成本、合规），该技能选择 LiteLLM / OpenRouter / Portkey 并生成路由配置。

## 练习

1. 运行 `code/main.py`。触发中断场景；确认故障转移落到第二个提供商且成本归属正确。

2. 添加语义缓存：提示词的 SHA256 作为查询键；缓存命中即时返回。测量重复调用上的成本节省。

3. 添加一个提示词分类器，将 `"code ..."` 提示词路由到偏好智能的别名，将 `"summarize ..."` 提示词路由到偏好速度的别名。

4. 设计按团队预算：每个团队有月度支出上限；网关一旦上限用尽就拒绝请求。选择一个执行粒度（按请求还是时间窗口）。

5. 并排阅读 LiteLLM、OpenRouter 和 Portkey 文档。说出每个平台独有而另外两个没有的功能。

## 关键术语

| 术语 | 大家怎么说的 | 实际含义 |
|------|----------------|------------------------|
| 路由网关 | "LLM 代理" | 位于多个提供商前、统一 API 表面的层 |
| OpenAI 兼容 | "讲 OpenAI schema" | 接收 `/v1/chat/completions` 形状，翻译为任意后端 |
| 模型别名 | "our_smart_model" | 代码中的名称，网关将其映射到具体模型 |
| 故障转移链 | "重试列表" | 失败时尝试的有序提供商列表 |
| 语义缓存 | "提示词 Embedding 缓存" | 键是提示词的 Embedding；近乎重复的提示词共用缓存命中 |
| 护栏 | "输入 / 输出过滤器" | PII 脱敏，拒绝策略违规 |
| 按密钥速率限制 | "团队预算" | 作用域为 API 密钥的配额 |
| 成本追踪 | "每请求支出" | Token 使用量 × 每模型单价 的聚合 |
| LiteLLM | "开源代理" | 可自托管的 OSS 路由网关 |
| OpenRouter | "托管 SaaS" | 基于积分计费的托管网关 |
| Portkey | "生产级选项" | 内置护栏的开源 + 托管方案 |
