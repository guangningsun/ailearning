# 原生稀疏注意力（DeepSeek NSA）

> 在 64k token 时，注意力占解码延迟的 70-80%。每个开放模型实验室都有一个修复计划。DeepSeek 的 NSA（ACL 2025 最佳论文）是那个最终被采纳的方案：三个并行注意力分支——压缩的粗粒度 token、有选择保留的细粒度 token、以及用于局部上下文的滑动窗口——通过一个可学习的门组合。它是硬件对齐的（内核友好）、原生可训练的（可在预训练中使用，而不是在推理时附加上去），在 64k 解码上比 FlashAttention 更快，同时匹配或优于全注意力质量。本课端到端构建三个分支，并展示为什么这种稀疏性是端到端可微分的。

**类型：** 构建型
**语言：** Python（标准库）
**前置条件：** 第 7 阶段 · 12（KV 缓存、flash-attention）、第 7 阶段 · 15（注意力变体）、第 10 阶段 · 16（差分注意力）
**时间：** 约 60 分钟

## 学习目标

- 陈述三个 NSA 注意力分支以及每个分支捕获什么。
- 解释为什么 NSA 是"原生可训练的"，而之前的稀疏注意力方法仅在推理时可用。
- 计算 NSA 与全注意力在 64k 上下文下作为压缩块大小和选择 top-k 函数的注意力计算节省。
- 在短合成序列上用标准库 Python 实现三分支组合，并验证门控权重行为。

## 问题

在序列长度 N 时，全注意力花费 `O(N^2)` 时间和每层 `O(N)` KV 缓存。在 64k token 时，计算和内存带宽数字是灾难性的。NSA 论文的测量理论估计：注意力在 64k 时占总解码延迟的 70-80%。下游的一切——TTFT、tokens/秒、每百万 token 成本——都由注意力成本主导。

稀疏注意力是明显的答案。之前的尝试分为两类。固定模式稀疏（滑动窗口、跨步、块局部）扔掉信息，在远程召回任务上失败。推理时稀疏（KV 缓存剪枝、H2O、StreamingLLM）应用于在稠密注意力上预训练的模型，只能恢复潜在加速的一小部分，因为模型从未被要求通过稀疏模式路由信息。

原生稀疏注意力（Yuan 等，DeepSeek + PKU + UW，ACL 2025 最佳论文，arXiv:2502.11089）同时做两件事：一个在预训练期间模型学习的稀疏模式，作为内核对齐算法实现，在推理时实际提供计算节省。两年后，NSA 或其直接后代将成为每个前沿长上下文模型的默认注意力。

## 概念

### 三个并行分支

对于每个 query，NSA 运行三次注意力，针对 KV 缓存的三个不同视图：

1. **压缩分支。** Token 被分组为大小为 `l`（通常为 32 或 64）的块。每个块通过一个小型学习 MLP 压缩成单个摘要 token。Query 在这些压缩 token 上运行注意力，获得整个序列的粗粒度视图。

2. **选择分支。** 使用来自压缩分支的注意力分数，识别与当前 query 最相关的 top-k 块。从那些块中读取细粒度（未压缩）token，query 在所有这些上运行注意力。将压缩分支注意力视为选择的路由信号。

3. **滑动窗口分支。** Query 关注最近 `W` 个 token（通常为 512）以获取局部上下文。这个分支捕获结构密集的短程模式（句法、局部共指），其他两个分支可能会错过。

三个分支输出通过一个对每个位置的小型学习 MLP 门组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是来自 query 上小型 MLP 的门权重。它们不必和为 1——它们可以独立地对分支加权。

### 为什么这是"原生可训练的"

选择步骤（top-k 块）是离散的。离散操作破坏梯度流。之前的稀疏注意力工作要么跳过通过选择的反向传播（限制训练），要么使用在推理时不能提供真正稀疏性的连续松弛。

NSA 巧妙地避开了这个问题：压缩分支注意力是整个序列上可微分的粗粒度注意力。Top-k 操作只是重用压缩分支中的 top 注意力分数来选择哪些细粒度块加载。梯度通过压缩分支分数流动（影响压缩输出和选择逻辑），所选块对最终输出的贡献也是可微分的。不可微分的 `top_k` 操作在前向计算图中是一个无操作——它只控制哪些块从内存加载。

这就是为什么 NSA 可以在预训练中端到端使用。模型学会通过三个分支联合路由信息，产生一个在推理时实际提供承诺加速的稀疏模式。

### 硬件对齐的内核

NSA 的内核是为现代 GPU 内存层次结构设计的。内核按 GQA 组（外循环）加载 query，按组（内循环）获取相应的稀疏 KV 块，并在 SRAM 上运行注意力。因为每个 query 组看到相同的所选块（选择是按 query 组而非按 query 头），KV 加载在组之间被摊销。算术强度保持高位。

论文报告在 64k 解码上 Triton 内核比 FlashAttention 快 9 倍，速度提升比例随序列长度增长。前向和后向内核都提供了。

### 计算预算

设 `N` 为序列长度，`l` 为压缩块大小，`k` 为 top-k 选择数量，`w` 为滑动窗口，`b` 为所选块大小（通常等于 `l`）。

- 压缩分支：每个 query `O(N/l)` 个 key，因此总共 `O(N * N / l)`。
- 选择分支：每个 query `O(k * b)` 个 key，因此总共 `O(N * k * b)`。
- 滑动分支：每个 query `O(w)` 个 key，因此总共 `O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

使用 `N = 64k, l = 64, k = 16, b = 64, w = 512`：每 query 成本为 `1000 + 1024 + 512 = 2536` 个 key。全注意力是 `64000` 个 key。25 倍计算减少。

使用 `N = 128k, l = 64, k = 16, b = 64, w = 512`：每 query 成本为 `2000 + 1024 + 512 = 3536` 个 key。全注意力是 `128000` 个 key。36 倍减少。收益随序列长度增长，这正是关键所在。

### 与其他方法比较

| 方法 | 可微分 | 真实推理加速 | 远程召回 |
|--------|---------------|----------------------|-------------------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 跨步/块稀疏 | 是 | 是 | 部分成功 |
| KV 剪枝（H2O、StreamingLLM） | N/A（推理时） | 是 | 部分成功 |
| MoBA（Moonshot） | 部分 | 是 | 良好 |
| NSA | 是（原生） | 是（64k 上 9 倍） | 匹配全注意力 |

MoBA（Moonshot，arXiv:2502.13189）同期发表，采用类似的三优于一的理念，将 MoE 原则应用于注意力块。NSA 和 MoBA 是 2026 年长上下文预训练需要知道的两个架构。

## 构建它

`code/main.py` 在短合成序列上实现三个分支，并展示：

- 压缩 MLP（为清晰起见使用简单的平均池化 baseline；真正的 NSA 使用学习到的 MLP）。
- 由压缩分支分数驱动的 top-k 块选择。
- 在最后 `w` 个 token 上的滑动窗口注意力。
- 门控组合。
- 与全注意力比较的计算计数打印。

### 第 1 步：将 token 压缩成块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 第 2 步：压缩分支注意力

对压缩后的 key 运行 query 的 softmax 注意力。压缩分支分数兼作 top-k 选择信号。

### 第 3 步：top-k 块选择

选取得分最高的 k 个压缩块的索引。从那些块加载原始未压缩的 token，并在它们上运行注意力。

### 第 4 步：滑动窗口注意力

取最后 `w` 个 token 并对它们运行标准注意力。

### 第 5 步：门控 + 组合

对 query 运行小型 MLP 产生三个门权重。最终输出是三个分支输出的加权和。

### 第 6 步：计算计数

打印每个分支每个 query 关注的 key 数和总数。与 `N`（全注意力）比较。在 `l = 32, k = 4, w = 128` 的 1024 token 合成上，NSA 每个 query 看 `32 + 128 + 128 = 288` 个 key 对比全注意力的 1024——3.5 倍更少。

## 使用它

NSA 正在 DeepSeek 自己的长上下文预训练流水线中发货。截至 2026 年 4 月公共推理栈中的集成状态：

- **DeepSeek 内部**：原生，发布的权重使用 NSA 或其后继者 DSA（Deepseek 稀疏注意力）。
- **vLLM**：正在为 DeepSeek-V3.x 权重开发实验性 NSA 支持。
- **SGLang**：发布了 NSA 基准；生产路径跟随 vLLM。
- **llama.cpp / CPU**：不支持；内核分解的开销在 CPU 吞吐下不值得。

何时使用 NSA：

- 针对 64k 及以上上下文且有严肃计算预算的预训练或继续训练运行。
- DeepSeek 自己的长上下文检查点的推理。权重是 NSA 原生的。

何时不使用：

- 服务一个已有的稠密注意力预训练模型。您无法在不继续训练的情况下改装 NSA。
- 上下文低于 16k。三分支开销主导节省。
- 批大小为 1 的交互式聊天。延迟敏感的解码有益，但只在长上下文上。

## 关键术语

| 术语 | 大家怎么说的 | 实际含义 |
|------|----------------|------------------------|
| 压缩分支 | "粗粒度视图" | 对块平均 key 的注意力，在每个 query O(N/l) 个 key 下提供全局上下文 |
| 选择分支 | "Top-k 块" | 对具有最高压缩分支分数的 `k` 个块的细粒度注意力 |
| 滑动窗口 | "局部上下文" | 对最后 `W` 个 token 的注意力，用于短程模式 |
| 原生可训练性 | "在稀疏性开启的情况下预训练" | 稀疏模式在预训练期间学习，不是附加在推理时 |
| 压缩块大小 l | "粗粒度视图的组大小" | 多少 token 合并成一个摘要；典型为 32-64 |
| Top-k | "保留的块" | 其未压缩 token 被读取的压缩块数量；典型为 16 |
| 滑动窗口 W | "局部注意力半径" | 通常为 512；更短伤害局部一致性，更长浪费计算 |
| 分支门 | "如何混合三个" | 对三个分支贡献加权的每位置 MLP 输出 |
| 硬件对齐 | "内核友好的稀疏性" | 稀疏模式被如此选择，使得实际 GPU 内核实现理论加速 |
| DSA | "NSA 的后继者" | Deepseek 稀疏注意力，NSA 之后 DeepSeek 系列中的后续架构 |
