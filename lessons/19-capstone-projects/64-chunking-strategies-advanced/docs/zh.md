# 分块策略对比

> 分块决定了检索器最终能返回什么。如果边界切得不对，无论用多好的 embedding 模型、重排序器还是 LLM，都无法修复下游的损伤。

**类型：** 构建型
**语言：** Python
**前置条件：** 阶段 11 第 04 课（embedding）、第 06 课（RAG）、第 07 课（高级 RAG）；阶段 19 Track B 基础（第 20-29 课）
**时间：** 约 90 分钟

## 学习目标

- 从零实现五种分块策略：固定窗口、句子、递归切分、语义聚类、结构化 Markdown 标题。
- 在带有答案标注范围的 fixture 语料上测量 recall@k，并解释为什么在散文类文档上一种策略胜出而在技术文档上却是另一种。
- 阅读分块长度分布，识别每种策略注入的失败模式：孤立句子、符号中途切断、仅有标题的分块、语义漂移。
- 通过检查三个属性（文档类型、平均段落长度、格式是否携带显式结构）为新语料选择默认策略，而无需运行基准测试。

## 问题

每个 RAG 流水线首先将源文档切分成小块——小到 embedding 模型能容纳，大到每个块包含一个自包含的想法。在哪里切并不是一个超参数。它决定了检索器最终能返回什么的上限。

一个查询问"预算中止阈值是什么样的"，只有当中止阈值所在的块能被检索到时才能成功。如果固定窗口分割器把阈值数值从周围上下文中切开了，embedding 就会跑到另一个聚类，BM25 分数下降，重排序器看到的是噪声，LLM 生成的答案就是错的。2024 年的论文《LongRAG: Enhancing Retrieval-Augmented Generation with Long-context LLMs》仅通过分块选择就测量到了 35% 的绝对检索召回率波动。2025 年的后续工作"上下文分块标题"缩小了差距，但没有消除它。

本课构建五种策略并排运行，在带有答案标注的 fixture 语料上测试，让你直接阅读召回数字。

### 固定窗口

暴力的基线方法。每 N 个字符切一刀。可选重叠，使得在位置 N 被切断的句子完整出现在起始位置为 N - overlap 的块中。速度快、确定性高，但在边界处理上很糟糕。作为对照使用，不要作为默认选项。

### 句子

用正则表达式或简单的状态机在句子边界处切分。将一个或多个句子打包成一个块，直到达到目标字符数。不在词中间切断。但仍在段落内部和章节内部切断。这是许多早期 RAG 流水线的默认选项，对于没有其他结构的散文来说是合理的选择。

### 递归切分

2023 年左右流行起来的层级策略。首先尝试最强分隔符（双换行、段落），回退到下一个（单换行），再到句子，最后到字符。递归终止条件是块符合预算。对于结构不一致的文档很强，因为它能自适应每个区域。

### 语义聚类

对每个句子做 embedding。将共享主题质心的连续句子聚在一起。每当与质心的运行相似度降到阈值以下时切一刀。边界反映的是语义而非字符。构建速度较慢，依赖于 embedding 模型，但对于文档内部段落切换主题的情况有很好的适应性。

### 结构化 Markdown 标题

对于带有显式结构的文档（Markdown、reStructuredText、RFC 风格编号章节），在标题边界处切分。每个块由标题及其下方所有内容组成，直到同级或更高级别的下一个标题。每个主题的块最小，但只有在语料格式规范时才可用。

### Recall@k 如何衡量边界选择

带标注的查询在源文档中携带答案片段的确切字符偏移量。分块后，你问：检索器返回的前 k 个块中是否有任何一个与标注答案重叠？如果有，该查询的 recall@k 为 1。如果没有，则为 0。在查询集上取平均。对每种策略运行相同评估，差异会告诉你哪种边界策略在你的语料上表现更好。

## 构建

`code/main.py` 实现了：

- `fixed_window(text, size, overlap)` - 基线方法。
- `sentence_chunks(text, target)` - 简单的句子打包器。
- `recursive_split(text, separators, target)` - 层级递归。
- `semantic_chunks(text, similarity_threshold)` - 基于确定性 mock embedding 的质心聚类。
- `structural_markdown(text)` - 标题感知的分割器。
- `mock_embed(text, dim)` - 基于哈希的 embedding，使代码可以离线运行。
- `DenseIndex` - 与阶段 19 Track B 混合检索课中相同的结构。
- `eval_recall(strategy, corpus, queries, k)` - 比较循环。
- `main()` - 在 fixture 语料上运行每种策略并打印 recall@k 表格。

运行：

```bash
python3 code/main.py
```

输出是一张小表格，每行一种策略，每列一个 k 值。句子在结构化 fixture 上表现不佳。结构化 Markdown 在 Markdown fixture 上胜出。递归在混合 fixture 上保持自己的水平，因为递归能自适应。语义聚类在散文 fixture 上胜出，因为那里没有有用的结构线索。

## 表格不会隐藏的失败模式

**孤立句子。** 句子打包产生的块会漏掉主题句子。embedding 随后指向错误的聚类。

**符号中途切断。** 固定窗口在代码或 YAML 中会把一个标识符切成两半。两半 embedding 出来的都是噪声。

**仅有标题的分块。** 结构化 Markdown 发出只包含 `## 标题` 的块。过滤掉这些，或者把下一个块的第一段附加上去。

**语义漂移。** 语义聚类在语料主题均匀时会切得太碎。一个 5000 字符的块把许多具体答案打包进一个分散的 embedding。把语义方法与硬字符上限结合。

**过时的 embedding。** 语义聚类使用 embedding 模型。如果你换了模型，分块也会变。将分块模型与检索模型分开固定，或者一起重建索引。

## 不运行基准测试选择默认策略

三个属性决定新语料的默认分块器。

| 属性 | 值 | 默认 |
|----------|-------|---------|
| 文档类型 | 无结构的散文 | 递归切分，目标 800 |
| 文档类型 | Markdown / RFC / API 文档 | 结构化 Markdown |
| 文档类型 | 代码 | AST 感知（超出范围；见阶段 19 第 02 课）|
| 段落长度 | 长，单一主题 | 句子，目标 500 |
| 段落长度 | 短，混合主题 | 语义，阈值 0.6 |

如有疑问，选择递归切分。它是最强的单一策略基线。

## 使用

生产模式：

- 在上线新流水线之前运行评估；不要信任你使用的库的默认策略。
- 每次更换 embedding 模型或语料混合时重新运行评估；胜者取决于语料。
- 在每个块的元数据中持久化策略名称，以便之后归因回归。

## 练习

1. 添加第六种策略：使用 `tiktoken` 的 token 窗口，而不是字符计数。在相同的 fixture 上与固定窗口对比。
2. 向散文 fixture 中注入 30% 的代码块。重新运行表格。解释为什么除了结构化 Markdown 之外每种策略都丢失了召回率。
3. 用你项目中真正的 provider 的 embedding 替换确定性 embedding。测量语义聚类召回率的变化。报告策略之间的差异是扩大了还是缩小了。
4. 为每个块添加一个 `summary` 字段：一句话的质心描述。将摘要附加到块正文重新运行评估。测量召回率的提升。

## 关键术语

| 术语 | 大家怎么说 | 实际含义 |
|------|-----------------|------------------------|
| Recall@k | "我们找到正确的块了吗？" | 前 k 个块中任意一个与标注答案重叠的查询比例 |
| 分块重叠 | "滑动窗口" | 将前一个块的最后 N 个字符包含到下一个块中 |
| 结构化分割器 | "标题感知的块" | 在 H1/H2/H3 边界处切割；标题文本是块的一部分 |
| 语义分块器 | "主题感知的块" | 对句子做 embedding，按质心相似度聚类，在漂移时切断 |
| 质心漂移 | "主题切换" | 运行均值与下一个句子之间的余弦相似度降到阈值以下 |
