# 使用 HTN 和进化搜索进行规划

> 符号规划处理计划在构造上可证明正确的场景。进化代码搜索处理适应度函数可机器检验的场景。ChatHTN (2025) 和 AlphaEvolve (2025) 展示了与 LLM 配对后各自释放了什么。

**类型：** 搭建
**语言：** Python（标准库）
**前置条件：** 阶段 14 · 02 (ReWOO 和 Plan-and-Execute)
**时间：** 约 75 分钟

## 学习目标

- 解释层次化任务网络：任务、方法、算子、前置条件、效果。
- 描述 ChatHTN 的混合循环 —— 带 LLM 后备分解的符号搜索。
- 解释 AlphaEvolve 的进化循环及其为何仅在有程序化评估器时有效。
- 用标准库实现一个玩具 HTN 规划器加一个玩具进化搜索。

## 问题

ReWOO（课程 02）、Plan-and-Execute 和 ReAct 涵盖了大多数智能体规划。有两种情况它们处理得不好：

1. **需要可证明正确性的计划。** 调度、航线规划、合规工作流 —— 计划必须在构造上合理。一个有时会幻觉步骤的流畅 LLM 计划是不可接受的。
2. **适应度函数可机器检验的优化。** 矩阵乘法、调度启发式、编译器通道 —— 目标不是"一个正确的计划"而是"最好的计划"。

HTN 规划和 AlphaEvolve 解决两个不同的问题。两者都将 LLM 作为放大器使用，而非替代品。

## 概念

### 层次化任务网络

HTN 包含：

- **任务** —— 复合的（待分解）和原始的（直接可执行）。
- **方法** —— 将复合任务分解为子任务的方式，带前置条件。
- **算子** —— 带前置条件和效果的原始动作。
- **状态** —— 一组事实。

规划：给定一个目标任务和初始状态，找到一个分解为满足前置条件的原始算子序列。

HTN 比 LLM 更早出现，至今仍是可证明正确计划的参考。

### ChatHTN (Gopalakrishnan et al., 2025)

ChatHTN (arXiv:2505.11814) 将符号 HTN 与 LLM 查询交织：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，问 LLM："在状态 `s` 中如何分解 `task`？"
3. 将 LLM 响应翻译为候选子任务。
4. 针对算子 schema 验证；拒绝无效分解。
5. 递归。

论文的核心主张：生成的每个计划都是可证明合理的，因为 LLM 建议仅作为候选分解进入，从不作为直接计划编辑。符号层拥有正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续）添加了一个学习器，通过回归泛化 LLM 产生的分解 —— 将 LLM 查询频率削减高达 75%。

### AlphaEvolve (Novikov et al., 2025)

AlphaEvolve (arXiv:2506.13131, DeepMind, 2025 年 6 月) 是另一种东西：由 Gemini 2.0 Flash/Pro 组合编排的进化代码搜索。

循环：

1. 从种子程序 + 程序化评估器（返回适应度分数）开始。
2. LLM 组合提出变异。
3. 通过评估器运行变异。
4. 保留最好的；再次变异。

发表的成果：

- 56 年来首次改进 Strassen 4×4 复数矩阵乘法（48 个标量乘法）。
- 通过 Borg 调度启发式回收 0.7% 的 Google 算力。
- 在前沿工作负载上实现 32% 的 FlashAttention 加速。

硬约束：适应度函数必须可机器检验。对散文答案的进化搜索不会收敛。

### 何时使用哪个

| 问题类别 | 使用 | 为什么 |
|---------------|-----|-----|
| 带硬约束的调度 | HTN + ChatHTN | 可证明的合理性 |
| 编译器优化 | AlphaEvolve | 机器可检验的适应度 |
| 多步任务执行 | ReAct / ReWOO | LLM 在循环中，无形式保证 |
| 有测试的代码改进 | AlphaEvolve | 测试即评估器 |
| 策略约束自动化 | HTN | 前置条件编码策略 |

### 这个模式会出问题的地方

- **没有算子的 HTN。** 没有前置条件/效果 schema，正确性主张就崩溃。ChatHTN 的"LLM 建议分解"需要 schema 来拒绝无效动作。
- **没有真实评估器的 AlphaEvolve。** "问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性的和快速的。
- **过度工程。** 大多数智能体任务两者都不需要。先用 ReAct 或 ReWOO。

## 搭建它

`code/main.py` 实现两个玩具：

- 一个带算子、方法、前置条件、效果的标准库 HTN 规划器，和一个 `LLMFallback`（当没有方法匹配复合任务时触发）。"LLM"是一个脚本化的分解器，所以规划器可以离线运行。
- 一个在算术程序上的标准库进化搜索：生长表达式，其输出在一组测试上最小化 `|f(x) - target|`。评估器是确定性的。

运行它：

```
python3 code/main.py
```

追踪显示 HTN 规划器分解一个复合任务（中途有 LLM 后备）和进化循环在目标表达式上收敛。

## 使用它

- **HTN 规划器** —— `pyhop`、`SHOP3`，或为特定领域策略执行构建自己的。
- **ChatHTN** —— 研究代码；该模式（符号 + LLM 后备）可移植到任何 HTN 规划器。
- **AlphaEvolve** —— DeepMind 论文；该模式（组合 + 评估器）可复现。OpenEvolve 和类似的开源分支正在出现。
- **智能体框架** —— 尚无一流 HTN 或 AlphaEvolve。作为子智能体或后台工作器构建。

## 交付它

`outputs/skill-hybrid-planner.md` 生成一个混合规划器脚手架（HTN 或进化），LLM 角色明确限定。

## 练习

1. 为 HTN 规划器添加回溯：当算子的后条件在运行时失败时，回滚并尝试下一个方法。
2. 给 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 中分解任务 `T` 时，存储结果。在下一次调用时先检查方法库。
3. 将进化搜索评估器换为真实测试套件。进化一个通过 20 个测试用例的排序函数；报告收敛代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 结合两者：用 HTN 将复合任务分解为子任务，然后在每个子任务的原始算子上使用进化搜索。它在何处闪光，在何处过度工程？

## 关键术语

| 术语 | 大家怎么说的 | 实际含义 |
|------|----------------|------------------------|
| HTN | "层次化规划器" | 带算子、前置条件、效果的任务分解 |
| 方法 | "分解规则" | 将复合任务拆分为子任务的方式 |
| 算子 | "原始动作" | 带前置条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 符号规划器在没有方法匹配时询问 LLM |
| AlphaEvolve | "进化代码搜索" | 组合 LLM 变异代码；确定性评估器选择 |
| 适应度函数 | "评估器" | 对输出的确定性、机器可检验的评分 |
| 在线方法学习 | "缓存 LLM 分解" | 存储并泛化 LLM 计划以削减查询成本 |
