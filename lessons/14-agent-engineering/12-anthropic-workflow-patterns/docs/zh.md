# Anthropic 工作流模式：简单优于复杂

> Schluntz 和 Zhang（Anthropic，2024 年 12 月）区分了工作流（预定义路径）和智能体（动态工具使用）。五种工作流模式覆盖了大多数场景。从直接调用 API 开始。只有在步骤无法预测时才引入智能体。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 阶段 14 · 01（智能体循环）
**时间：** 约 60 分钟

## 学习目标

- 说出 Anthropic 的五种工作流模式：提示链、路由、并行化、编排器-工作者、评估器-优化器。
- 解释智能体与工作流的区别，以及每种方案的工程成本。
- 判断何时该选工作流、何时该选智能体（反之亦然）。
- 使用标准库在脚本化 LLM 上实现所有五种模式。

## 问题

团队在只需要一个函数调用的问题上往往会动用多智能体框架。代价是真实的：框架增加了层层抽象，遮蔽了提示词、隐藏了控制流、引入了过早的复杂性。Schluntz 和 Zhang 在 2024 年 12 月发表的文章是业界引用最多的反思：从简单开始，只有当复杂性值得其成本时才引入。

## 概念

### 工作流与智能体

- **工作流。** 通过预定义的代码路径编排 LLM 和工具。工程师拥有这张图。
- **智能体。** LLM 动态指挥自己的工具、采取自己的步骤。模型拥有这张图。

两者各有其用武之地。工作流更便宜、更快、更容易调试。智能体解锁了开放式问题，但也让失败模式更难推理。

### 增强型 LLM

五种模式的基础：具有三项能力的单个 LLM —— 搜索（检索）、工具（动作）、记忆（持久化）。任何 API 调用都可以使用这些能力。

### 五种模式

1. **提示链。** 调用 1 的输出是调用 2 的输入。当任务有清晰的线性分解时使用。各步骤之间可以有编程式的门控。
2. **路由。** 一个分类器 LLM 选择调用哪个下游 LLM 或工具。当输入需要不同处理方式时使用（一级支持 vs 退款 vs 缺陷 vs 销售）。
3. **并行化。** 并发运行 N 个 LLM 调用，聚合结果。两种形态：分块（不同块）和投票（相同提示词，N 次运行，取多数或综合）。
4. **编排器-工作者。** 一个编排器 LLM 动态决定运行哪些工作者（也是 LLM），并综合它们的输出。类似于智能体循环，但编排器不会无限循环。
5. **评估器-优化器。** 一个 LLM 提出答案，另一个 LLM 评估它。迭代直到评估通过。这是 Self-Refine（第 05 课）的泛化形式。

### 工作流胜过智能体的地方

- **可预测的任务。** 如果你能枚举步骤，就应该枚举。
- **有成本约束的任务。** 工作流的步骤数有上限；智能体可能失控。
- **有合规约束的任务。** 审计员希望看到图，而不是从轨迹中推断。

### 智能体胜过工作流的地方

- **开放式研究。** 下一步取决于上一步返回了什么。
- **可变长度任务。** 从几分钟到几小时的工作，步骤数未知。
- **新领域。** 当你尚不知道正确的工作流时 —— 先探索，再固化。

### 上下文工程配套学科

"有效的 AI 智能体上下文工程"（Anthropic，2025 年）形式化了相邻的学科：200k 窗口是一笔预算，而非容器。包含什么、何时压缩、何时让上下文增长。在阶段 14 关于上下文压缩的课程中有详细阐述（本课程重新编号前为阶段 14 第 06 课）。

## 构建

`code/main.py` 针对 `ScriptedLLM` 实现了所有五种工作流模式：

- `prompt_chain(input, steps)` —— 顺序执行。
- `route(input, classifier, handlers)` —— 分类 + 分发。
- `parallel_vote(prompt, n, aggregator)` —— N 次运行，聚合。
- `orchestrator_workers(task, workers)` —— 编排器选择工作者。
- `evaluator_optimizer(task, proposer, evaluator, max_iter)` —— 循环直到通过。

运行：

```
python3 code/main.py
```

每种模式都会打印其执行轨迹。每种模式的代码约 10-15 行；引入框架的成本以千行计。

## 使用

- 大多数任务用直接 API 调用。
- 只有当模式真正需要持久状态（LangGraph）、参与者模型并发（AutoGen v0.4）或角色模板（CrewAI）时才引入框架。
- 当你想要 Claude Agent SDK 的形态但不想重新构建时，使用它。

## 练习

1. 实现带置信度阈值的路由。低于阈值 -> 升级给人工。对于一级支持场景，阈值应该设在哪里？
2. 给 `parallel_vote` 添加超时。当一个调用挂起时会发生什么？如何聚合有缺失的投票？
3. 把 `evaluator_optimizer` 变成一个多臂老虎机：保留跨迭代的前 2 名输出，这样后期的好结果不会被后期的坏结果覆盖。
4. 将提示链与路由结合：一个路由器在三条链中选一条。测量 token 成本与单一超大提示词方案的对比。
5. 选一个你的生产特性。画出工作流图。统计步骤数。这里用智能体真的更好吗？

## 关键术语

| 术语 | 大家怎么说的 | 实际含义 |
|------|----------------|------------------------|
| 工作流 | "预定义流程" | 工程师所有的 LLM 和工具调用图 |
| 智能体 | "自主 AI" | 模型所有的图；动态工具指挥 |
| 增强型 LLM | "带工具的 LLM" | LLM + 搜索 + 工具 + 记忆；原子单位 |
| 提示链 | "顺序调用" | 调用 N 的输出是调用 N+1 的输入 |
| 路由 | "分类器分发" | 选择由哪条链或模型处理输入 |
| 并行化 | "扇出" | N 个并发调用；按分块或投票聚合 |
| 编排器-工作者 | "调度智能体" | 编排器 LLM 动态选择专家 LLM |
| 评估器-优化器 | "提议者 + 裁判" | 迭代直到评估通过；Self-Refine 的泛化 |
